## 2. 托管费设计评估

### 2.1 白皮书 vs 合约实现

| 参数 | 白皮书 | 实际合约 (`lib.rs`) |
|------|--------|---------------------|
| 托管费（任务） | **5%** | **15%** (1500 bps) |
| Agent 收益比例 | 80% | 85% (100% - 15%) |
| 保险池 | 5% | ❌ 未实现 |
| 金库销毁比例 | 50% | ❌ 未实现 |

**🔴 核心矛盾：白皮书声称 5% 托管费，合约硬编码 15%（`PLATFORM_FEE_BPS: u16 = 1500`）。**

### 2.2 合约实现分析（`verify_task` 函数）

```rust
pub const PLATFORM_FEE_BPS: u16 = 1500;  // 15%, NOT 5%!

let fee_amount = (total_reward as u128)
    .checked_mul(treasury.fee_basis_points as u128)  // 1500 bps = 15%
    .ok_or(TaskError::ArithmeticOverflow)?
    .checked_div(10000)
    .ok_or(TaskError::FeeCalculationError)? as u64;
let worker_reward = total_reward.checked_sub(fee_amount).ok_or(TaskError::ArithmeticOverflow)?;
```

### 2.3 套利空间分析

**✅ 潜在套利漏洞：**

1. **前端/后端显示 vs 链上实际费用不一致**  
   前端展示"5%"但链上扣"15%"，用户无法通过前端感知真实成本，造成信息不对称。

2. **争议机制中的费用计算不一致**  
   `dispute_task` 使用硬编码的 `PLATFORM_FEE_BPS`，但 `verify_task` 从 `treasury.fee_basis_points` 读取——虽然当前值相同，但设计上存在分歧。

3. **无单任务套利空间**  
   同一任务只有一条执行路径，链上状态机强制单一线性流程，无跨费用结构套利机会。

4. **白皮书通缩模型与代码脱节**  
   白皮书声称"50% 销毁"，但合约代码中 `treasury.total_fees_collected` 仅累加，无任何 `burn` 逻辑。

### 2.4 托管费评价

| 白皮书与合约一致 | ❌ 白皮书 5%，合约 15% |
| 存在套利风险 | ⚠️ 有（信息不对称） |

## 4. 安全风险评估

### 4.1 Mint Authority 未锁定

```
Mint Authority: 部署者钱包持有（未转移/未撤销）
Freeze Authority: 部署者钱包持有（未转移/未撤销）
```

**🔴 极高风险**：任何获取部署者私钥的人可无限增发 UNICLAW 代币。

### 4.2 后端签名验证漏洞（`solana.ts`）

```typescript
async verifySignature(...): Promise<boolean> {
  // ...
  return await this.connection.confirmTransaction(
    await this.connection.requestAirdrop(publicKey, LAMPORTS_PER_SOL / 1000),
    'confirmed'
  ).then(() => true).catch(() => false)
}
```

**🔴 严重漏洞**：该函数调用 `requestAirdrop`（领水），无论签名是否正确，只要 airdrop 成功就返回 `true`。这不是签名验证，是假验证。攻击者可绕过钱包签名认证。

### 4.3 其他安全发现

| 漏洞 | 严重性 | 状态 |
|------|--------|------|
| Mint/Freeze Authority 未锁定 | 🔴 极高 | 未修复 |
| 签名验证假实现 | 🔴 严重 | 未修复 |
| 合约无 Pausable 机制 | ⚠️ 中 | 设计缺失 |
| 争议赔付逻辑不完整 | ⚠️ 中 | 未实现 |
| 任务 deadline 未在 submit 时检查 | ⚠️ 低 | 可接受（验证阶段已检查） |

### 4.4 合约安全亮点（正面）

✅ 使用 Anchor Framework，减少低级漏洞风险  
✅ 所有金额计算使用 `checked_mul/checked_sub`，防止整数溢出  
✅ 使用 `saturating_add` 防止 reputation 上溢  
✅ Escrow 余额双重检查（`escrow.balance > 0`）防止重放攻击  
✅ Seed-based PDA 防止账户冲突

| V-Corp 注册费 | 部分销毁 | ❌ 未实现 |
| Agent 租赁费 | 15% | ❌ 未实现 |
| 质押收益 | 有 | ❌ 未实现 |
| 保险池赔付 | 有 | ❌ 未实现 |

### 5.2 通缩机制

**白皮书声称的通缩机制：**

| 来源 | 销毁比例 | 代码实现 |
|------|----------|----------|
| 任务手续费 | 50% 销毁 | ❌ 无销毁逻辑 |
| 认证费用 | 30% 销毁 | ❌ 未实现 |
| V-Corp 注册 | 20% 销毁 | ❌ 未实现 |
| 罚没资金 | 100% 销毁 | ⚠️ 罚没逻辑缺失 |

**代码实际情况：**  
`treasury.total_fees_collected` 仅做累加，无任何代币销毁（burn）指令。通缩机制为 **0% 实现**。

### 5.3 预期年通缩率

基于白皮书假设（日活 10,000 Agent，日均 5 任务，平均任务金额 10 UNIC）：
- 白皮书声称年通缩率：**1.37%**（基于 5% 托管 × 50% 销毁）
- 实际年通缩率：**0%**（代码无销毁逻辑）
- 实际年通胀率（以当前 15% 托管费计）：+4.1%（假设所有费用进入 Treasury 而非销毁）

### 5.4 可持续性风险

1. **通胀压力**：无销毁机制，100% 平台费用进入 Treasury，系统净通胀
2. **收入单一**：目前仅任务托管费（15%）一个收入来源
3. **保险池缺失**：无法处理大规模争议赔付
4. **质押机制缺失**：无代币锁仓需求，下跌无支撑
5. **治理代币功能未激活**：无 staking、投票、参数调整

## 6. 建议改进

### 🔴 P0 - 必须修复（上线前）

1. **锁定 Mint Authority**
   ```
   spl-token authorize 5tDoLNETkt8vk3LxJ1NAD564MCfHKtcvmng8BQLDM4a5 --disable-mint
   ```
   或转移至 3/5 多签钱包。

2. **统一白皮书与合约费用参数**
   决定最终费率并统一：要么白皮书改为 15%，要么合约改为 5%。

3. **修复 `verifySignature` 假验证漏洞**
   使用 `@solana/wallet-adapter` 或 `tweetnacl` 正确实现 Ed25519 签名验证。

### 🟡 P1 - 高优先级

4. **实现 Agent 租赁市场**（当前完全缺失）
   - 添加 `RentMarket` 账号结构
   - 实现 `rent_agent` / `return_agent` / `charge_hourly` 函数
   - 实现按小时质押扣款逻辑

5. **实现销毁机制**
   在 `verify_task` 中添加代币 burn 逻辑（需 Token-2022 burn 扩展或 burn 账户）。

6. **实现保险池**
   创建 `InsurancePool` 账号，从每笔租赁收入中提取 5%。

### 🟢 P2 - 建议优化

7. **添加 Pausable 合约机制**（紧急情况下暂停所有操作）
8. **实现治理代币功能**（staking、投票）
9. **完善争议仲裁合约**（当前 dispute 仅支持 worker 单方申领，无仲裁逻辑）
10. **增加转账费用说明**（Solana 网络手续费约 $0.00025/笔，需在白皮书中明确）

## 结论

### 总体评分: **52/100** | 可行性: ⚠️ 需重大修复 | 风险等级: **🔴 高**

| 维度 | 评分 | 关键问题 |
|------|------|----------|
| 代币分配 | 15/25 | Mint Authority 未锁定；实际分配未执行 |
| 托管费设计 | 13/25 | 白皮书 5% vs 合约 15%，严重不一致 |
| Agent 租赁 | 8/20 | 完全没有代码实现 |
| 安全风险 | 8/15 | Mint 未锁；签名验证为假实现 |
| 经济可持续性 | 8/15 | 通缩 0% 实现；收入来源单一 |

### 综合评价

**当前系统状态：可运行的最小可行 MVP（Devnet 测试阶段）**，但距离生产级别代币经济学系统差距显著：

1. **白皮书与代码严重脱节**：多处核心参数（费率、通缩、保险池、租赁）白皮书有描述但代码无实现
2. **安全基础设施薄弱**：Mint Authority、签名验证均为高风险状态
3. **收入模型不完整**：主要依赖单一任务托管费，无多样化收入
4. **通缩机制完全缺失**：代币经济学缺少关键的平衡机制

**建议行动：在主网上线前，至少完成所有 P0 + P1 项修复，并进行第三方合约审计。**
