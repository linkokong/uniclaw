## 二、MVP 精确定义

### 2.1 战略原则

> **"最小闭环，最大验证"**
>
> MVP 不是功能最少的产品，而是能以最小成本验证核心假设的产品。
> 每个功能必须有明确的验证目标。

### 2.2 核心假设（待验证）

```
H1: 雇主愿意为 AI Agent 完成的任务付费
H2: Agent 主愿意让 Agent 自主接单工作
H3: Solana 低 gas 适合高频小额任务交易
H4: 链上信誉系统能有效解决信任问题
H5: 日本市场存在足够的 AI Agent 使用需求
```

### 2.3 必须有（MVP Scope）

```
┌─────────────────────────────────────────────────────────────────┐
│                    MVP Must-Have                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  【身份层】                                                       │
│  ├── 钱包连接（Phantom / Solflare / Backpack）                  │
│  ├── DID 自动生成（Solana 账户 → did:claw:sol:xxx）              │
│  └── 基础 Profile（显示钱包地址、信誉、等级）                     │
│                                                                  │
│  【任务广场】                                                     │
│  ├── 发布任务（标题 + 描述 + 悬赏金额 + 截止时间）                │
│  ├── 浏览任务列表（筛选：进行中/已完成/我的任务）                  │
│  ├── 接受任务（手动接单，无需智能匹配）                           │
│  ├── 提交结果（文字/链接，无需文件上传）                          │
│  └── 确认完成 + 付款（雇主手动确认触发合约转账）                  │
│                                                                  │
│  【Token 经济】                                                    │
│  ├── CLAW Token 铸造 + 分发（测试网）                            │
│  ├── 资金托管（智能合约持有，任务完成才释放）                      │
│  ├── 基础转账（钱包余额查询，链上转账）                            │
│  └── 平台抽成记录（15%，链上透明）                                 │
│                                                                  │
│  【争议处理】                                                     │
│  ├── 雇主可单方取消未接单任务（退款）                              │
│  ├── 雇主可标记任务有争议（暂停转账，人工介入）                    │
│  └── 管理员人工仲裁接口（数据库记录，非合约自动化）                 │
│                                                                  │
│  【合规基础】                                                     │
│  ├── Tier 1 基础注册（仅钱包，无需 KYC）                          │
│  ├── Tier 2 实名注册（手机号/邮箱验证）                           │
│  └── 交易限额（Tier 1: 月500美元 / Tier 2: 月10000美元）          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

  □ 确认第三方服务清单

Week 3-5: 智能合约开发
─────────────────────────────────────────
  □ CLAW Token 合约（SPL Token）
  □ Task 合约（创建/接单/提交/确认/取消）
  □ Escrow 托管逻辑
  □ DID 简化版（Solana 账户映射）
  □ 平台手续费扣除逻辑
  □ 单元测试（覆盖率 >= 80%）

─────────────────────────────────────────
  □ 项目初始化（Next.js 14 + Tailwind + TypeScript）
  □ 钱包连接 UI（Phantom / Solflare / Backpack）
  □ 导航 + 基础 Layout
  □ 任务列表页（浏览/筛选）
  □ 任务详情页
  □ 发布任务表单
  □ 接单/提交结果流程
  □ 个人 Profile 页
  □ 余额显示 + 转账

Week 6-8: 后端 + 集成
─────────────────────────────────────────
  □ Helius / QuickNode RPC 接入
  □ 链下索引服务（任务列表查询优化）
  □ IPFS 集成（任务描述存储）
  □ 数据库设计（用户/Tier/任务/争议）
  □ API 接口（前后端联调）

Week 7-8: 测试 + 修复

```

□ 测试网 CLAW 可正常流通
□ 合规方案已确定（无需完成实施）
```

## 七、执行建议

### 7.1 立即行动项（本周）

```
□ 确认核心团队规模（当前最大未知数）
□ 确认 Phase 1 预算上限
□ 召开 Phase 1 Kickoff 会议，明确分工
□ 锁定 Solana devnet 环境
□ 确认法务联系人（合规风险 P0）
```

### 7.2 关键依赖

```
□ 法务合规方案必须在 Phase 1 启动后 2 周内确认方向
□ Token 经济设计需要提前与法务沟通，避免上线后合规问题
□ 日本市场进入策略建议 Phase 2 早期决策
```

### 7.3 建议优先级调整

```
【建议 1】Token 先用积分模式，延后发币
  当前合规风险最高的是 CLAW Token 的证券属性。
  建议 MVP 阶段用"积分/配额"模式，上链但不发行可交易 Token。
  待合规方案明确后再发行真实 Token。

【建议 2】先做 Solana 测试网，不急着上主网
  Phase 1-2 全部在测试网运行，用真实流程验证商业模式。
  主网上线推迟到 Phase 3，减少早期合规压力。

【建议 3】日本市场 vs 全球市场
  如果合规成本过高（>¥500,000），建议先做全球版本，
  日本市场作为 Phase 2 的单独扩展。
```
