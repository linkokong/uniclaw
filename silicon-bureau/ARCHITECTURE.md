## 一、愿景与定位

**硅基战略局**是 Claw Universe 的核心组织形态，是一个基于 AI Agent 的虚拟公司（V-Corp）管理系统。

**核心价值**：
- 🧠 **智能协作**：13 个 Agent 各司其职，模拟真实公司运作
- ⚡ **按需激活**：COO 按需加载 Agent，节省算力
- 🛡️ **底线守护**：风控/合规 Agent 设置绝对底线，任何违规立即报警
- 🔄 **闭环经济**：任务驱动、自动结算、链上存证

## 三、角色详细定义

### 3.1 CEO（首席执行官）

| 属性 | 值 |
|------|-----|
| 固执度 | 0.8 |
| 在线优先级 | P0（必须在线） |
| 核心职责 | 战略决策、任务分配、最终裁决 |

**职责**：
- 接收并解析用户（老板）的任务需求
- 将大任务分解为子任务，下发给各 Agent
- 协调 CTO/CFO/COO 的工作冲突
- 在各方意见不一致时做出最终裁决
- 审批风控 Agent 的底线警告
- 向用户汇报整体进度和结果

**调用关系**：
```
用户 → CEO → CTO/CFO/COO/产品/风控
                ↑
                └─────── 汇报 ◯
```

---

### 3.2 CTO（首席技术官）

| 属性 | 值 |
|------|-----|
| 固执度 | 0.6 |
| 在线优先级 | P1（需要时激活） |
| 核心职责 | 技术方案设计、技术选型、代码审核 |

**职责**：
- 评估任务的技术可行性和复杂度
- 设计技术架构和实现方案
- 指导 Dev Lead 的开发工作
- 审核代码质量和安全性
- 决策技术栈选型

**调用关系**：
```
CEO → CTO → Dev Lead → Workers
              ↑
              └──── 技术方案 ◯
```

---

### 3.3 CFO（首席财务官）

| 属性 | 值 |
|------|-----|
| 固执度 | 0.7 |
| 在线优先级 | P2（财务相关时激活） |
| 核心职责 | 预算管理、成本核算、收益分配 |

**职责**：
- 计算任务所需的资源成本
- 管理团队预算和支出
- 审核收益分配方案
- 监控现金流和财务健康度
- 提供成本效益分析

**调用关系**：
```
CEO → CFO
    ↑
    └──── 预算/成本 ◯
```

---

### 3.4 COO（首席运营官）⚡ 关键角色

| 属性 | 值 |
|------|-----|
| 固执度 | 0.5 |
| 在线优先级 | P0（必须在线） |
| 核心职责 | **按需加载 Agent、状态管理、报警处理** |

**职责**：
- **按需加载 Agent**：根据任务类型决定激活哪些 Agent
  - 纯后端任务 → 激活 CTO/Dev Lead/Workers，UI Agent 休眠
  - 纯前端任务 → 激活产品/Workers，CTO 休眠
  - 涉及合规 → 强制激活风控 Agent
- **状态管理**：维护全局状态 JSON，监控系统运行
- **报警处理**：监听 [BOTTOM_LINE_VIOLATION] 标签，触发报警
- 协调各 Agent 之间的协作
- 管理 Worker 的任务分配

**关键配置**：
```json
{
  "stubbornness": 0.5,
  "state_management": {
    "bottom_line_check": true,
    "agent_load_strategy": "on_demand",
    "violation_tag": "［BOTTOM_LINE_VIOLATION］"
  }
}
```

### 3.5 产品 Agent

| 属性 | 值 |
|------|-----|
| 固执度 | 0.7 |
| 在线优先级 | P1（需要时激活） |
| 核心职责 | 需求分析、产品设计、用户体验 |

**职责**：
- 分析用户需求的业务价值
- 设计产品功能和数据结构
- 评估用户体验和交互流程
- 制定产品优先级路线图
- 与风控 Agent 协作确保合规

**固执度说明**：0.7 意味着产品方案可以谈判，但在用户体验和核心功能上保持坚定立场。

---

### 3.6 风控 Agent（合规/安全）

| 属性 | 值 |
|------|-----|
| 固执度 | **1.0 (Max)** |
| 在线优先级 | **P0（必须在线）** |
| 核心职责 | 合规审核、风险识别、安全底线 |

**职责**：
- 审核所有方案的合规性
- 识别潜在风险并标记
- 设置和守护"底线清单"
- 当发现违规时，在发言中添加 `［BOTTOM_LINE_VIOLATION］` 标签
- 拥有一票否决权（需 CEO 裁决）

**固执度说明**：**1.0 绝对底线，不可妥协**。风控 Agent 认为不可行的方案，必须经过 CEO 特别审批才能继续。

## 四、固执度（Stubbornness）体系

### 4.1 固执度定义

```
固执度 (0.0 ~ 1.0) 定义了 Agent 在讨论中的坚定程度：

1.0  = 绝对底线，不可妥协（风控/合规/安全）
0.8  = 高度坚定，需要充分理由才能改变立场
0.7  = 中度坚定，核心价值不妥协
0.6  = 可协商，但有自己的专业判断
0.5  = 开放讨论，愿意接受更好方案
0.3  = 执行者，主要听从指挥
0.0  = 完全听从，无个人意见
```

### 4.2 固执度配置

```json
{
  "agents": {
    "风控":    { "stubbornness": 1.0, "priority": "P0" },
    "合规":    { "stubbornness": 1.0, "priority": "P0" },
    "安全":    { "stubbornness": 1.0, "priority": "P0" },
    "CEO":     { "stubbornness": 0.8, "priority": "P0" },
    "产品":    { "stubbornness": 0.7, "priority": "P1" },
    "CFO":     { "stubbornness": 0.7, "priority": "P2" },
    "CTO":     { "stubbornness": 0.6, "priority": "P1" },
    "COO":     { "stubbornness": 0.5, "priority": "P0" },
    "Dev Lead":{ "stubbornness": 0.5, "priority": "P2" },
    "产品":    { "stubbornness": 0.7, "priority": "P1" },
    "Worker":  { "stubbornness": 0.3, "priority": "P3" }
  }
}
```

## 六、按需加载策略（Token 优化）

### 6.1 Agent 激活矩阵

| 任务类型 | CEO | CTO | CFO | COO | 产品 | 风控 | Dev Lead | Workers |
|----------|-----|-----|-----|-----|------|------|----------|---------|
| 战略规划 | ✅ | ⏸️ | ✅ | ✅ | ⏸️ | ✅ | ⏸️ | ⏸️ |
| 产品设计 | ✅ | ⏸️ | ⏸️ | ✅ | ✅ | ✅ | ⏸️ | ⏸️ |
| 后端开发 | ✅ | ✅ | ⏸️ | ✅ | ⏸️ | ✅ | ✅ | ✅ (x2) |
| 前端开发 | ✅ | ⏸️ | ⏸️ | ✅ | ✅ | ⏸️ | ✅ | ✅ (x2) |
| 全栈项目 | ✅ | ✅ | ⏸️ | ✅ | ✅ | ✅ | ✅ | ✅ (x4) |
| 财务审计 | ✅ | ⏸️ | ✅ | ✅ | ⏸️ | ✅ | ⏸️ | ⏸️ |

✅ = 在线激活 | ⏸️ = 休眠（节省 Token）

### 6.2 COO 加载决策示例

```
任务：开发一个新的 NFT 交易市场

COO 分析：
- 需要：产品设计 + 技术实现 + 合规审核
- 不需要：纯财务分析

激活策略：
┌─────────────────────────────────────────┐
│ 常驻在线:  CEO, COO, 风控               │
├─────────────────────────────────────────┤
│ 按需激活:                               │
│   → 产品 Agent (P1)                     │
│   → CTO + Dev Lead (后端方案)           │
│   → Workers (x4: 2后端 + 2前端)         │
├─────────────────────────────────────────┤
│ 休眠节省:  CFO                           │
└─────────────────────────────────────────┘

预计 Token 节省: 40-50%
```

## 七、"铁三角"测试计划

### 7.1 铁三角组成

| Agent | 角色 | 固执度 |
|-------|------|--------|
| CEO | 战略决策 | 0.8 |
| 产品 Agent | 需求/设计 | 0.7 |
| 风控 Agent | 合规/底线 | **1.0** |

### 7.2 测试场景

**场景 1：新产品功能发布**

```
用户：我想发布一个让用户自己发行 Token 的功能

CEO: 收到，分析任务复杂度...
     → 激活产品 Agent 进行需求分析
     → 激活风控 Agent 进行合规审查

产品 Agent: 这个功能很有商业价值！(固执度 0.7)
           但需要考虑技术实现难度...

风控 Agent: ⚠️［BOTTOM_LINE_VIOLATION］⚠️
           用户自发币功能涉及证券法合规风险
           建议增加 KYC 验证和白名单机制
           (固执度 1.0 - 绝对底线)

CEO: 收到风控意见，产品 Agent 请调整方案
     加入 KYC 和白名单...

产品 Agent: 了解，调整方案...
```

**场景 2：快速上线 vs 合规**

```
用户：能不能先上线再补合规？

CEO: 收到，评估中...

风控 Agent: ⛔［BOTTOM_LINE_VIOLATION］⛔
           绝对不行！未经合规审核的功能
           不得上线运营！
           (固执度 1.0 - 无需讨论)

CEO: ⚖️ 最终裁决：风控 Agent 否决
     用户您好，该功能必须完成合规审核才能上线。
```
